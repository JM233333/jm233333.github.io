关于时间和空间复杂度的估算

关于时间复杂度

通常情况下，时限 111 秒所能支撑的运算次数在 1e81e81e8 次左右，而且这还是常数不能太大并且运行环境的性能较高的情况，否则可能就会卡常数。如果运算次数在 1e71e71e7 次左右是非常稳妥的，不需要担心TLE，除非你代码写丑了。当然，具体的运算速度还是取决于实际编译环境。

关于空间复杂度

通常情况下，内存限制所能支撑的最多是开上几个大小为 1e61e61e6 的数组，如果比这更大的话，最好自己估算一下内存大小，以防MLE。当然，有些题目的内存限制故意设得很小，或是内存给得很大，这就要另外考虑了。

另外，多数OJ的系统栈空间都不是很大，因此需要尽量避免进行超大深度的递归，这可能导致RE或MLE，因而难以准确判断出错原因。

关于取模

在许多竞赛题中，因为答案太大，而出题方又并不打算考察大整数或高精度的运算，就要求输出答案对某个数取模的结果。例如，求斐波那契数列的第 nnn 项 (n≤1e18)(n \le 1e18)(n≤1e18) ，答案对 1e9+71e9 + 71e9+7 取模。

显然这是一个非常好的运算规则，这意味着我们不需要求出最终答案后再取模，而是可以在计算过程中不断取模，以避免使用大整数。

需要注意的是除法模运算不满足该规则，当出现除法时，千万不能想当然地去做。除法取模需要用到数论中的逆元，此处不予赘述。

关于数组的一个使用技巧

在C++中对局部变量的大小限制要远胜于对全局变量的大小限制。在做竞赛题时数组大小通常是很大的，所以我们通常会把数组声明为全局变量，这样可以避免无意义的CE或RE，也可以避免因为自己的不小心而导致的TLE（比如在函数里声明一个比较大的数组，然后程序又需要反复调用这个函数，导致大量的内存操作）。

有一点需要注意，局部变量的数组如果不初始化就是未定义行为（乱码），而全局变量的数组是默认会初始化为全 000 的。

此外，我们通常会把数组开得比最大值要大一些，例如数据范围为 n≤1000n \le 1000n≤1000 ，那么我们可以开一个大小为 100510051005 的数组。这样可以方便我们用下标 1..n1..n1..n 进行操作。

这什么意思呢？在有些题目中，思维或者是实现过程比较复杂，并且你会发现按照计算机的习惯用 0..n−10..n-10..n−1 的下标会让你非常头疼。比如一个图上节点编号从 111 到 nnn ，输入格式都是这样的，你按照 0..n−10..n-10..n−1 就不得不给每个编号都 −1-1−1 ，算完之后输出的时候还得 +1+1+1 ，非常麻烦。还有一种情况是比如给定的递推式就是从 000 到 nnn 的，而你的数组大小为 nnn 就不够用，还得特殊处理初值。

关于多组数据的输入输出

可以做HDU 1089~1096这八道题来练习多组数据的输入输出。

首先回答一个常见的疑问，有些人会想，因为多组数据之间存在这个输入的问题，我们是不是需要把每组数据的对应的答案都保存下来，处理完所有数据之后再把它们一起输出，否则OJ会不会判我错误。

关于cin/cout和scanf/printf

因为某些原因（详情自己百度），cin/cout的效率要远低于scanf/printf，因此我们建议大家尽量不要使用。

如果不得不使用（例如你想直接读入一个std::string），那么可以使用一些骚操作。该语句的作用是关闭cin/cout与scanf/printf的同步，可以使其效率变得可以接受，但是你就不能使用scanf/printf了，否则会触发未定义行为。

关于float和double

因为竞赛中的浮点数运算经常有精度要求，而float的精度很低，在很多情况下都是不够用的，所以索性就不要去使用它，用double。

在一些毒瘤题目中，可能double的精度都不足（当然，有时候是因为你的代码写的不好），这时可以尝试使用long double。不过经验告诉我，在多数情况下，都是因为你的代码实现方式的精度不足，只有少数情况是因为题目本身毒瘤。

关于一些头文件

因为各处编译环境并不相同，所以有时你的代码在本机跑得好好的，或是在别的OJ上跑得好好的，但是交上去就是会CE。这里给出一些常见的头文件使用建议：

在使用max、min、sqrt……之类的函数时，建议把cmath和algorithm头文件带上。

在POJ上不要对long long类型使用max、min库函数，他会给你当成int用，一定要自己写。

在使用memset、memcpy、strcmp、strcpy……之类的函数时，建议把cstring头文件带上。

在使用sort、reverse……之类的函数时，建议把algorithm头文件带上。

在使用greater、less……之类的内置类型时，建议把functional头文件带上。

在使用pair、tuple……之类的内置类型时，建议把utility头文件带上。
