基本概念

二叉查找树 (Binary Search Tree, BST) 是一种追加了特殊限制的二叉树。它对二叉树上的每个节点追加了一个保证，对于任意一个节点上的值p->value，我们保证：

  1. 如果 ppp 的左子树非空，那么对于左子树上的任意一个节点 qqq ，都保证p->value > q->value。

  2. 如果 ppp 的右子树非空，那么对于右子树上的任意一个节点 qqq ，都保证p->value < q->value。

下图是一棵二叉查找树的例子。

二叉查找树也可以递归地定义，根据上述定义，显然每个节点的左子树和右子树也都是二叉查找树。

为便于理解，我们先假设树上不存在等值的节点，并在后文中给出处理多个相同元素的方法。

原理

二叉查找树相比于二叉树仅仅追加了两条约束，但这两条约束是具有重大价值的。

考虑一棵普通的二叉树，每个节点上存储了一个值，如果你希望查找二叉树上是否存在某个值 xxx ，那么你只能穷竭搜索整棵树上的所有节点，无论是什么存储方式都是一样。也就是说，在二叉树上查找某个值 xxx 是否存在的时间复杂度为 O(n)O(n)O(n) 。

当然，在线性表中查找某个值 xxx 是否存在的时间复杂度也是 O(n)O(n)O(n) ，不过这在很多情况下是可以设法改进的，不过这不是本文的讨论范围。

事实上，二叉查找树的这一限制，会使得查找某个值 xxx 是否存在的期望时间复杂度达到 O(log⁡n)O(\log{n})O(logn) ，这显然是一个巨大的改进。二叉查找树的核心原理就在于它的查找，无论是插入还是删除操作都是由查找衍生而成的。下面我们将对查找的原理给出详细解释。

首先，在平均情况下，一棵节点个数为 nnn 的二叉树的高度是 O(log⁡n)O(\log{n})O(logn) 的。这一点从完全二叉树的高度就能看出，此处不予赘述。

其次，我们考虑如何进行查找。我们从根节点出发开始递归查找，设当前节点为 ppp 。

  1. 如果x == p->value，那么查找成功。

  2. 如果x < p->value，又已知p->value小于 ppp 的右子树上的任意值，这就意味着 ppp 的右子树上不可能存在一个节点的值为 xxx ，所以我们只需要在左子树上继续进行查找，递归左子树即可。

  3. 如果x > p->value，又已知p->value大于 ppp 的左子树上的任意值，这就意味着 ppp 的左子树上不可能存在一个节点的值为 xxx ，所以我们只需要在右子树上继续进行查找，递归右子树即可。

  4. 如果p == nullptr，表明查找失败。当然，4.要在1.之前进行判断。

递归过程是不断向下的，也就是说，我们最多递归 O(h)O(h)O(h) 次，其中 hhh 为树的高度，那么在平均情况下查找的时间复杂度为 O(log⁡n)O(\log{n})O(logn) 。

为便于理解，我们以上一小节中的图示为例，举两个查找的例子。

对于 x=4x = 4x=4 ：

  1. 来到值为 555 的根节点，因为x = 4 < 5 = p->value，所以答案肯定不可能在右子树，令p = p->left。

  2. 来到值为 333 的节点，因为x = 4 > 3 = p->value，所以答案肯定不可能在左子树，令p = p->right。

  3. 来到值为 444 的节点，发现x = 4 = 4 = p->value，查找成功。

对于 x=6x = 6x=6 ：

  1. 来到值为 555 的根节点，因为x = 6 > 4 = p->value，所以答案肯定不可能在左子树，令p = p->right。

  2. 来到值为 777 的节点，因为x = 6 < 7 = p->value，所以答案肯定不可能在右子树，令p = p->left。

  3. 发现p == nullptr，查找失败。

不知你是否注意到，我们一直强调的是平均情况，因为树的结构并不总是接近、或至少和完全二叉树相差不太远的。对于一棵退化的二叉查找树，树的结构是近似或者就是一条链的。而如果我们又查找的是某个叶子节点处的值，那么查找的时间复杂度将会退化为 O(n)O(n)O(n) 。也就是说，在最坏情况下查找的时间复杂度为 O(n)O(n)O(n) 。

下图中给出了一棵退化的二叉查找树的例子，如果我们不停地查找 x=5x = 5x=5 ，那么每次查找都是 O(n)O(n)O(n) 的。

插入操作

二叉查找树的插入是不会破坏树上已有节点的结构的，对于任意的数值，在被插入后都将会成为树上的叶子节点，这一点需要牢记。这样的规定是为了插入的方便，毕竟胡乱插入的话也没有意义。

插入操作事实上是基于查找的，我们只需先在二叉查找树中查找待插入的值，如果找到，则说明已存在，不需要重复插入（当然，如果你希望支持重复元素存在的话，用第一小节所述的方法即可）。否则，直到查找失败为止，那么这一空位就是该数值应当插入的位置，将其插入为叶子节点。

删除操作

要想删除指定的某个数值，显然我们需要先查找到它的位置，如果查找失败自然就不需要删除了。

如果待删除节点是一个叶子节点，那么问题很好解决，但它还可能是非叶子节点，我们必须对此进行处理。总之，删除操作比插入操作要复杂不少。

为便于描述，我们设 ppp 为待删除节点， fpf_pfp​ 为 ppp 的父节点， lp, rpl_p,\ r_plp​, rp​ 为 ppp 的左右子节点。

对删除操作的处理可以分三种情况讨论：

  1. 待删除节点是叶子节点。

此时直接将节点删除即可。

  1. 待删除节点只有一个子节点。

此时将 lpl_plp​ 或 rpr_prp​ 连同以该子节点为根节点的子树一起上移，用 lpl_plp​ 或 rpr_prp​ 代替 ppp 的位置即可。

我们证明这不会破坏二叉查找树的性质，这很简单。假设 ppp 是 fff 的左子节点（是右子节点同理可证），那么以 ppp 为根节点的子树上的所有节点上的值都比 fff 上的值要小，因此将以 lpl_plp​ 或 rpr_prp​ 为根节点的子树代替原先以 ppp 为根节点的子树，都不会破坏 fff 的性质。而 lpl_plp​ 或 rpr_prp​ 的性质本就未受影响，因此以这样的策略删除 ppp 是合理的。

注意在代码实现时不能直接 delete q ，在前面更基础的教程中应当已经提到过类似的问题了。

  1. 待删除节点有两个子节点。

这种情况则有些复杂，但也并没有那么困难，只是相对复杂而已。

因为存在两个子树 lpl_plp​ 和 rpr_prp​ ，我们不能像第二种情况那样直接挪移上去。我们考虑从这两棵子树中找到一个合适的节点代替 ppp 的位置，使得二叉查找树的性质不变。找到思路就容易了，显然可以选择左子树中的最大值或者右子树中的最小值。

以左子树中的最大值为例，记最大节点为 mmm 。首先，因为 mmm 是选定的最大值，所以 mmm 必然大于以 lpl_plp​ 为根节点的子树上的所有节点上的值。其次，因为 mmm 原本属于左子树，所以 mmm 必然小于以 rpr_prp​ 为根节点的子树上所有节点上的值。因此用 mmm 替代 ppp 后二叉查找树的性质不变。右子树中的最小值同理，二者等价，凭个人喜好选择即可。

根据二叉查找树的基本原理，左子树中的最大节点 mmm ，要么是叶子节点，要么只有左子节点。因为如果 mmm 有右子节点，则它的右子树上一定存在更大的节点。因此在将 mmm 移走时，只需执行类似于处理删除操作的情况1和情况2的处理方式即可。

在实际的代码实现中，我们有如下的一些技巧可以精简代码。

  1. 对于情况1和情况2，在代码实现时，情况1可以被归入情况2内，因为用 ppp 的空左子树 lpl_plp​ 来替换 ppp 事实上等价于将 ppp 直接删除，本来就是令 p = nullptr 。

  2. 对于情况3，在用左子树最大值 mmm 来代替当前节点 ppp 时，为减少复杂的指针操作，我们直接将 mmm 中的数值替换掉 ppp 中的数值，然后递归去将节点 mmm 真正删除。也可以直接将节点 ppp 删除然后让 mmm 取代 ppp ，但那样需要一些比较复杂的指针操作，既然可以避免，为什么不呢。

处理重复元素

之前我们假设树上不存在等值的节点，现在我们考虑如何处理多个相同的元素。有两种主要的方法。

用多个节点来存储副本

其实很简单，只要将二叉查找树的判断条件 x < p->value 和 x > p->value 改为如下两组之一即可：

  1. 判断条件 x <= p->value 和 x > p->value 。

  2. 判断条件 x < p->value 和 x >= p->value 。

但是不能使用x <= p->value 和 x >= p->value ，学完二叉查找树你应该能发现，如果这样设置判断条件，当 x == p->value 时，我们究竟应该插入到左子树还是右子树？这在逻辑上就有很大的问题。

这种解决方案非常简单粗暴，但是存在较大的弊端，例如难以支持一些高级查询操作，也难以对某个元素的出现次数进行计数。

在节点上添加统计信息

我们为每个节点额外增加一个成员变量 cntcntcnt ，用于记录该元素值的出现次数，这可以省去很多麻烦，且易于扩展。

在删除操作中，如果在查找过程中发现存在一个节点满足 value == p->value ，则应当修改 cntcntcnt 值，而不是执行删除动作。

应用

二叉查找树的一个重要性质是，对一棵二叉查找树进行中序遍历，可以直接得到一个有序的升序序列。这个性质只要知道就很简单，自己想一下就明白了。

但是除此以外，因为退化的二叉查找树的性能很差，各类操作的时间复杂度可以退化到 O(n)O(n)O(n) ，和在一个普通的线性表上操作一样低效，而且二叉查找树还非常容易发生退化，比如进行一系列有序元素的插入。

因此，普通的二叉查找树缺乏实用价值。幸运的是，平衡树能够很好地解决二叉查找树的退化问题，此处不予赘述，详见下一篇文档。
