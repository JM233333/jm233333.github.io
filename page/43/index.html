前言

这里是关于我的文档内容的一些规范和解释。

我的所有文档均使用C++语言。

我的文档被划分为若干个大的知识板块，在每个知识板块内，每篇文档会有一个标号，这个标号表示了该文档大致的在当前知识板块内的相对级别，建议按标号从小到大阅读，标号相同的则无所谓。

有些文档会具有相同的标号，这是按层划分的，可能会让您感到有些困惑，您可以参阅本文底部的阅读指引图来获得确切的阅读顺序。

一些文档的开头会给出前置知识声明，如果你没有掌握这些知识，建议进行递归学习。这些前置声明和阅读指引图提供的顺序是相同的。

每篇文档的开头均包含版权声明和对本文适用人群的声明，分为“面向初学者”和“面向进阶读者”两大类。

面向初学者：

  1. 适合对当前的大的知识板块掌握尚浅的读者。注意，是相对于当前知识板块，而不是相对于所有知识。

  2. 默认读者没有接触过任何相关知识，会对所有涉及到的、读者可能不知道的知识进行解释。

  3. 通常会以大量的篇幅对简单的基础知识进行解释，用语言引导读者举一反三深入思考。

  4. 会使用精简代码并分段提取出来讲解，用大量的简单示例图来帮助学习，完整的代码实现也会附于文末。

  5. 因为是面向初学者的，哪怕是非常简单的内容，文中也会花费大量篇幅进行解释，因而通常不适合基础较好的读者，请自行避雷。

面向进阶读者：

  1. 适合对当前的大的知识板块已经有所了解，希望学习该板块中更高级的内容的读者。

  2. 默认读者的基础较为良好，不会花费大量篇幅解释一些我认为过于简单的内容。

  3. 通常会深入解释知识的内在原理，甚至给出正确性证明等，帮助读者完全掌握知识，而非浮于表面。

  4. 会提供更多课本上没有的扩展知识，并且通常只给出完整的代码实现，除非是一些重要或较难的关键代码会分段提取出来讲解。

  5. 有些是专门针对ACM选手的文章，知识难度大且实用价值没有那么高，此类情况会特别标注在文章的开头，不打算参加算法竞赛的同学可以自行避过。

特别声明，我的文章并不能帮助你通过学校的书面考试，但能帮助你真正地掌握这些知识，能让你学会如何用代码实现这些基础算法与数据结构。如果只是为了应付考试，那么我的文章并不适合你。

关于时间和空间复杂度的估算

关于时间复杂度

通常情况下，时限 111 秒所能支撑的运算次数在 1e81e81e8 次左右，而且这还是常数不能太大并且运行环境的性能较高的情况，否则可能就会卡常数。如果运算次数在 1e71e71e7 次左右是非常稳妥的，不需要担心TLE，除非你代码写丑了。当然，具体的运算速度还是取决于实际编译环境。

这里给出一个数据范围与时间复杂度的对照表，基准是 1000ms ~ 2000ms ，意思是某个数据范围 在多数情况下 能接受的最高的时间复杂度，是个人在ACM生涯中积累的经验。但是注意 这并不是绝对的 ，仅是 最常见 的可能性！

例如在一些跑得快的OJ上 n≤106n \le 10^6n≤106 也可以用 O(nlog⁡n)O(n\log{n})O(nlogn) AC，甚至 n≤104n \le 10^4n≤104 可以用 O(n2)O(n^2)O(n2) AC，而且并不到极度罕见的程度。另外如果代码常数很小，或是跑不满不会达到最坏情况，或是该题可以用剪枝手段来规避最坏情况，也很可能可以AC。因此切忌以此为标尺使用，仅仅心中有数即可。

关于空间复杂度

通常情况下，内存限制所能支撑的最多是开上几个大小为 1e61e61e6 的数组，如果比这更大的话，最好自己估算一下内存大小，以防MLE。当然，有些题目的内存限制故意设得很小，或是内存给得很大，这就要另外考虑了。

另外，多数OJ的系统栈空间都不是很大，因此需要尽量避免进行超大深度的递归，这可能导致RE或MLE，因而难以准确判断出错原因。

关于取模

在许多竞赛题中，因为答案太大，而出题方又并不打算考察大整数或高精度的运算，就要求输出答案对某个数取模的结果。例如，求斐波那契数列的第 nnn 项 (n≤1e18)(n \le 1e18)(n≤1e18) ，答案对 1e9+71e9 + 71e9+7 取模。

首先给出模运算的概念：

给定一个正整数 ppp ，任意一个整数 aaa ，一定存在等式 a = kp + ra = kp + ra = kp + r ，其中 k, rk,\ rk, r 是整数，且 0 ≤r < p0 \le r < p0 ≤r < p 。

称 kkk 为 nnn 除以 ppp 的商， rrr 为 nnn 除以 ppp 的余数。

对于正整数 ppp 和整数 a, ba,\ ba, b ，定义取模运算 a % pa \% pa % p （或 a mod   pa \mod pa mod p ），表示 aaa 除以 ppp 的余数，即 a%p=ra \% p = ra%p=r 。

关于数组的一个使用技巧

在C++中对局部变量的大小限制要远胜于对全局变量的大小限制。在做竞赛题时数组大小通常是很大的，所以我们通常会把数组声明为全局变量，这样可以避免无意义的CE或RE，也可以避免因为自己的不小心而导致的TLE（比如在函数里声明一个比较大的数组，然后程序又需要反复调用这个函数，导致大量的内存操作）。

有一点需要注意，局部变量的数组如果不初始化就是未定义行为（乱码），而全局变量的数组是默认会初始化为全 000 的。

此外，我们通常会把数组开得比最大值要大一些，例如数据范围为 n≤1000n \le 1000n≤1000 ，那么我们可以开一个大小为 100510051005 的数组。这样可以方便我们用下标 1..n1..n1..n 进行操作。

基本概念

什么是数据结构？

数据结构就是按照一定的方法去存储和维护计算机数据的一个抽象概念。这可能听起来有些令人头大，但它没那么难，比如说，其实一个普通的数组就是一种数据结构——从存储上讲，它以连续线性空间来存储数据；从维护上讲，它支持简单的随机访问和修改，但却难以在其中部位置进行插入和删除操作（你不得不先移动一大堆数据才能完成插入和删除）。

所以说，对于一种数据结构，其实你只需要关心两个问题：它如何存储数据？它如何维护数据？仅此而已。至于数据结构提供了哪些有用的功能，有什么优缺点，其实都是由这两个问题的答案决定的。

数据结构的存储问题，又可以分为物理结构和逻辑结构这两个问题。物理结构指的是数据在内存中的实际存储方式，亦称为存储结构。而逻辑结构指的是人们从主观上为其赋予的、语义上的存储方式。

物理结构和逻辑结构之间没有任何必然的联系。物理结构取决于实际是如何用代码实现数据结构的，如果不考虑数据结构的效率，设计者甚至可以随意地设计它的物理结构。而逻辑结构是数据结构本身的内涵，对于某一种数据结构而言，其逻辑结构应当是不变的，如果改变了的话就成了另外一种数据结构了。

物理结构可以分为顺序存储、链式存储、索引存储、散列存储四大类。或许你只接触过前两种，不过大多数基础数据结构也就是顺序或链式的，索引和散列相对更常见于操作系统、计算机组成、数据库等高年级课程中。

逻辑结构可以分为线性结构和非线性结构两大类，线性结构自不必说，非线性结构可以是树、可以是图。

至于大类之下还可以再细分，此处不予赘述。

数据结构的维护问题，通常包括它如何支持你对其中存储的数据的增删改查，即插入、删除、修改、查询。此外，对于一些高级数据结构，它还需要在你执行完增删改查操作后对其中存储的数据进行维护，可能包括重排、自适应调整、旋转等。

通常，每一种数据结构都会支持增删改查的操作，关键在于每一种操作在各种情境下的的效率如何，也就是时间复杂度的优劣。例如数组显然不适合需要频繁在表头处增删的情况，而链表显然不适合需要频繁进行随机查找的情况，把它们互换过来就很合适了。

深入理解

数据结构有什么用？

对于你曾经写过的那些简单程序而言，数据结构的确没什么用，似乎只用数组、结构体和类就能解决一切。因此很难和没有接触过数据结构的人去解释数据结构的用途，你现在只需要记住：它真的很重要。

大部分的教材都是直接把各种基础数据结构挨个灌进你的脑子，却从来没告诉过你，它们究竟是怎么来的。

首先要弄清楚的是，数据结构不是凭空产生的， 一种数据结构之所以会存在、之所以会被发明出来，是因为前人在实际应用中需要它们。 是因为先有了“待解决的问题”，才有了数据结构作为“解决方案”，而不是由数据结构的存在而衍生出各种问题。

以编程语言为例，是因为我们需要处理批量的数据，所以那些早期的编程语言的发明者才会制造出数组来。同样地，是因为我们有时需要动态开辟内存，才会有malloc/free和new/delete的存在，而不是因为它们存在，我们就可以动态开辟内存。它们的产生是源自于我们的实际需求的。

这些听起来似乎有些啰嗦和缺乏实际意义，但如果你没有理解这一点，多多少少会影响到你对数据结构的进一步学习和掌握。每一个数据结构的产生都是因为前人遇到了需要它们去解决的问题。或许现在的你听着有些懵，希望你在学习了一段时间以后能够理解我为什么会用这么大的篇幅解释这一点——只要你在学习每一个数据结构的时候，着重注意它们是如何被运用来解决实际问题的。

因此你要牢记：数据结构只是一种工具，就像编程语言也只不过是工具一样，编程的核心还是人的思想。你需要掌握并灵活运用这些数据结构，仅仅把它们背下来是没有意义的。
