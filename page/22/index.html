要求实现 next_permutation 函数的功能，即给定一个序列 numsnumsnums ，将该序列重排，得到字典序恰好比原序列大一位的新序列。如果当前序列为字典序最大，则将其重排为字典序最小。

题解

原题给出的样例太小，下面两个样例可以帮助我们进行思考：

为便于描述，设 numsnumsnums 的下标为 0,1,...,n−10, 1, ..., n-10,1,...,n−1 。

显然，当序列 numsnumsnums 非严格单调递减时，不存在比它字典序更大的重排序列。由此容易发现，当序列的后缀 nums[i+1, n−1]nums[i+1,\ n-1]nums[i+1, n−1] 非严格单调递减时，下标 iii 处应当以“最小的幅度”变大。这个变化就是与后缀中的某个元素交换，且后缀 nums[i+1, n−1]nums[i+1,\ n-1]nums[i+1, n−1] 将被重排为递增序列。至于前缀 nums[0, i−1]nums[0,\ i-1]nums[0, i−1] 则保持不变。

也就是说，我们需要找到最靠右的一组严格递增的相邻元素，即找到最大的 i∈[0, n−2]i \in [0,\ n-2]i∈[0, n−2] 满足 nums[i]>nums[i+1]nums[i] > nums[i+1]nums[i]>nums[i+1] 。这可以通过一次遍历实现。

然后，为了使下标 iii 处以“最小的幅度”变大，我们需要将 nums[i]nums[i]nums[i] 与后缀 nums[i+1, n−1]nums[i+1,\ n-1]nums[i+1, n−1] 中最小的比 nums[i]nums[i]nums[i] 大的元素交换，再将后缀重排为递增序列即可。时间复杂度为 O(nlog⁡n)O(n\log{n})O(nlogn) 。

题解

为便于描述，设 sss 的下标为 0,1,...,n−10, 1, ..., n-10,1,...,n−1 。

方法1：栈

许多括号匹配问题都可以用栈解决。我们依次遍历整个字符串，遇到左括号则将当前下标入栈，遇到右括号则出栈，并进行一些细节处理。

在遇到右括号并出栈后，如果栈不为空则更新答案，如果栈为空，则将当前下标入栈，这样在将来出栈时便于更新答案。

一次遍历即可解决问题，时间复杂度为 O(n)O(n)O(n) 。

方法3：思维

这个方法较难想到，我是从一个错误的思路出发才想到的。

如果你曾想过用双指针扫描解决本题，那么一定会想到初始令 left=0, right=−1left = 0,\ right = -1left=0, right=−1 ，然后令 rightrightright 递进并统计左右括号数，当右括号数多于左括号数时重置 left=right+1left = right + 1left=right+1 。

但是容易发现，当左括号数多于右括号数时，例如对于 ((()() ，当 right=3right = 3right=3 时，我们无法决定何时令 leftleftleft 递进并寻找合法解，存在 O(n)O(n)O(n) 种可能性，因为当 rightrightright 继续递进时可能发现新的右括号。这意味着单向扫描无法覆盖到所有答案，左括号数多于右括号数的情况无法处理，该方法似乎并不正确。

事实上，我们只需要倒着再做一遍，即可处理左括号数多于右括号数的情况，正反两遍扫描即可覆盖所有合法解。

总的时间复杂度为 O(n)O(n)O(n) 。该方法的时间常数比前两者更大，唯一的优势在于该方法仅需要 O(1)O(1)O(1) 的额外空间。
